# AGENTS.md — Codex Instructions for Tastytrade QQQ / NVDA / SOXL Bot

## 0. Project Identity

This repository is for a systematic options trading assistant focused on QQQ, NVDA, SOXL, SPY, IWM, and SMH.

The project goal is **not** to build an uncontrolled "auto profit bot". The goal is to build a professional research-first, dry-run-first trading system that can eventually support carefully gated small-size live trading.

Primary project documents:

1. `Tastytrade_QQQ_NVDA_SOXL_Bot_Implementation_Plan.md`
2. `Tastytrade_QQQ_NVDA_SOXL_Strategy_Spec_v2_Expert.md`
3. This file: `AGENTS.md`

When there is a conflict, follow this priority order:

1. Safety and risk-control rules in this `AGENTS.md`
2. Strategy rules in `Tastytrade_QQQ_NVDA_SOXL_Strategy_Spec_v2_Expert.md`
3. Engineering plan in `Tastytrade_QQQ_NVDA_SOXL_Bot_Implementation_Plan.md`
4. Local code comments and implementation details

## 1. Non-Negotiable Safety Rules

These rules must not be bypassed, softened, or removed without an explicit human instruction in the repository.

### 1.1 No Live Trading in Early Versions

Do not implement live order submission as the default behavior.

Allowed execution modes:

- `research`
- `backtest`
- `paper`
- `dry_run`

Live trading must remain disabled unless all of the following are true:

1. `ENABLE_LIVE_TRADING=true` is explicitly set in environment variables.
2. The code path requires an additional explicit runtime confirmation flag.
3. The order passes the portfolio risk engine.
4. The order is defined-risk.
5. The order is not 0DTE.
6. The order is logged before submission.
7. The account is not in kill-switch mode.

Default mode must be `dry_run`.

### 1.2 Forbidden Trade Types

The first production-capable version must reject:

- Naked short calls
- Naked short puts
- Undefined-risk positions
- 0DTE options
- Market orders for options
- Orders with missing max-loss calculation
- Orders with missing exit plan
- Orders with excessively wide bid/ask spreads
- Orders during configured hard-blocked event windows unless explicitly allowed by strategy config

### 1.3 Risk Engine Has Veto Power

The risk engine must be able to reject any trade, even if the strategy score is high.

Never let strategy code directly submit or dry-run an order without passing through the risk engine.

Required risk checks:

- Per-trade max loss
- Daily loss limit
- Weekly loss limit
- Total open max loss
- Symbol concentration
- Strategy concentration
- SOXL-specific cap
- Consecutive loss count
- Event risk block
- Liquidity block
- Kill-switch state

### 1.4 No Credentials in Code

Never hardcode usernames, passwords, account numbers, API keys, tokens, or OpenAI keys.

Use environment variables and `.env.example` only.

Required behavior:

- `.env` must be ignored by git.
- `.env.example` may contain placeholder keys only.
- Tests must use mock credentials.

### 1.5 LLM Cannot Control Live Strategy

The LLM reviewer may:

- Summarize trades
- Classify mistakes
- Generate candidate rules
- Produce weekly reports
- Recommend research tasks

The LLM reviewer must not:

- Modify live strategy parameters automatically
- Increase position size automatically
- Remove stop-loss logic
- Override the risk engine
- Submit orders
- Approve live trading

Candidate strategy changes generated by LLM must go into a research queue and require backtesting, out-of-sample testing, dry-run validation, and human review.

## 2. Technical Stack Preferences

Use Python-first architecture unless the human explicitly asks otherwise.

Recommended stack:

- Python 3.11+
- `pydantic` for schemas and validation
- `pandas`, `numpy` for data manipulation
- `SQLAlchemy` or `sqlmodel` for persistence
- SQLite for local development
- PostgreSQL optional for production
- `pytest` for tests
- `ruff` for linting
- `black` for formatting
- `mypy` for type checking where practical
- `FastAPI` for API layer, optional but preferred
- `uvicorn` for local API serving
- `typer` or `argparse` for CLI commands
- `python-dotenv` for local env loading

Broker/API integration:

- Use an adapter interface so the core system can run with mock data, CSV data, or tastytrade.
- Keep tastytrade SDK/API calls isolated under `src/broker/`.
- Do not let broker-specific models leak into strategy or risk modules.

## 7. Strategy System Requirements

Implement strategy logic from `Tastytrade_QQQ_NVDA_SOXL_Strategy_Spec_v2_Expert.md`.

### 7.1 Strategy Engines

Required engines:

1. Short Premium Income Engine
   - Put credit spread
   - Call credit spread

2. Neutral Range Engine
   - Iron condor

3. Trend Participation Engine
   - Call debit spread
   - Put debit spread

4. Calendar / Diagonal Engine
   - Calendar spread
   - Diagonal spread
   - Lower priority; only after core engines are stable

5. Hedge / Risk-Off Engine
   - No new short premium during crash regime
   - Optional defined-risk hedge candidates

### 7.2 Score-Based Entry

Do not require every condition to be perfect.

Use a score-based system:

```text
Total score = 100
- Regime fit: 30
- Volatility edge: 25
- Liquidity quality: 20
- Price action: 15
- Event risk: 10
```

Entry thresholds:

```text
80-100: high-quality trade, normal size
65-79: good trade, reduced or normal size
55-64: small experimental trade only
<55: no trade
```

The strategy engine may generate candidates, but the risk engine must approve them.

### 7.3 Regime Classifier

Implement a classifier with these regime labels:

- `bull_trend_low_mid_iv`
- `bull_trend_high_iv`
- `range_high_iv`
- `range_low_iv`
- `bear_trend_high_iv`
- `crash_risk_off`

Required inputs when available:

- QQQ close vs EMA20 / EMA50
- SPY close vs EMA20 / EMA50
- QQQ 5-day and 20-day returns
- SMH or SOXX semiconductor trend
- VIX level and change
- IV Rank or IV Percentile
- Realized volatility vs implied volatility
- Event calendar

When an input is unavailable, return a degraded-confidence regime decision rather than crashing.

## 8. Option Selection Rules

### 8.1 General Liquidity Filters

Reject option chains or contracts when:

- Missing bid or ask
- Bid/ask spread is too wide relative to mid
- Volume and open interest are too low
- Mid price cannot be computed
- Expiration is blocked
- DTE is below minimum

Configurable defaults:

```yaml
liquidity:
  max_bid_ask_pct_of_mid: 0.15
  min_open_interest: 100
  min_volume: 10
  allow_missing_greeks: false
```

Small-account settings may be slightly looser, but the system must log all liquidity warnings.

### 8.2 DTE Guidelines

Default DTE ranges:

```yaml
dte:
  short_premium_min: 21
  short_premium_max: 60
  neutral_range_min: 30
  neutral_range_max: 45
  trend_qqq_nvda_min: 14
  trend_qqq_nvda_max: 45
  trend_soxl_min: 7
  trend_soxl_max: 21
  calendar_front_min: 7
  calendar_front_max: 21
  calendar_back_min: 30
  calendar_back_max: 60
  forbidden_dte_min: 1
```

0DTE is forbidden in the first production-capable version.

### 8.3 Delta Guidelines

Defaults:

```yaml
delta:
  short_premium_min_abs: 0.10
  short_premium_max_abs: 0.35
  iron_condor_short_min_abs: 0.16
  iron_condor_short_max_abs: 0.25
  trend_long_min_abs: 0.45
  trend_long_max_abs: 0.65
  trend_short_min_abs: 0.20
  trend_short_max_abs: 0.40
```

## 10. Order Building and Execution

All orders must go through this pipeline:

```text
StrategyCandidate
  -> RiskEngine.evaluate(candidate, portfolio_state)
  -> OrderBuilder.build(candidate)
  -> BrokerAdapter.dry_run(order)
  -> TradeLogger.record(order, risk_decision, dry_run_result)
```

Live path, when eventually enabled:

```text
StrategyCandidate
  -> RiskEngine.evaluate(candidate, portfolio_state)
  -> OrderBuilder.build(candidate)
  -> BrokerAdapter.dry_run(order)
  -> Human/live gate check
  -> BrokerAdapter.submit(order)
  -> TradeLogger.record(order, live_result)
```

Never call broker submit directly from strategy code.

Use limit orders for options.

Prefer mid-price based limit logic, with configurable offsets:

```yaml
execution:
  credit_spread_limit_offset_from_mid: 0.02
  debit_spread_limit_offset_from_mid: 0.02
  max_price_chase_attempts: 2
  price_chase_increment: 0.01
```

## 11. Backtesting Requirements

Backtesting must include realistic assumptions:

- Bid/ask spread
- Slippage
- Commission and fees if available
- Partial fill assumptions where possible
- Event windows
- DTE decay
- Expiration handling
- Early exit rules

Do not report only win rate.

Required metrics:

- Total return
- CAGR if period supports it
- Max drawdown
- Profit factor
- Win rate
- Average win
- Average loss
- Average win / average loss
- Expectancy per trade
- Sharpe, if suitable
- Sortino, if suitable
- Calmar, if suitable
- Exposure time
- Number of trades
- Consecutive losses
- Worst trade
- Worst day
- Worst week

A strategy candidate must not be promoted from research to dry-run unless it passes configured minimum thresholds.

## 13. Testing Requirements

Every important module needs tests.

Minimum tests:

### 13.1 Risk Engine Tests

Test that the risk engine rejects:

- Naked options
- 0DTE
- Missing max loss
- Too much same-symbol exposure
- Too much total open max loss
- Too large SOXL trade
- New trades after daily loss limit
- New trades after max consecutive losses
- Market orders for options

### 13.2 Strategy Tests

Test that strategy scoring:

- Produces high score for correct regime fit
- Produces lower score for poor liquidity
- Penalizes event risk
- Does not crash on missing IV data
- Rejects score below threshold

### 13.3 Order Builder Tests

Test:

- Put credit spread legs are correct
- Call credit spread legs are correct
- Call debit spread legs are correct
- Put debit spread legs are correct
- Iron condor has correct four-leg structure
- Max profit and max loss calculations are correct

### 13.4 Broker Adapter Tests

Test with mock broker:

- Dry-run does not submit live orders
- Submit is disabled unless live flag is enabled
- Failed broker responses are logged
- Missing credentials fail safely

## 18. Design Principles

Prefer:

- Explicit risk rules over hidden assumptions
- Small modules over large scripts
- Typed schemas over loose dictionaries
- Config-driven parameters over hardcoding
- Mockable broker interfaces over direct API calls everywhere
- Deterministic tests over manual inspection
- Audit logs over console-only output
- Dry-run first, live trading last

Avoid:

- Strategy logic inside broker adapters
- Risk logic inside strategies
- Silent failures
- Live trading as default
- One-off notebooks as production code
- Unbounded retries
- Market orders for options
- Any code path where LLM can alter production risk limits
